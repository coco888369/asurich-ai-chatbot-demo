## 技術スタック（概要）

### 構成
- **本番**: PHPベースのAPIプロキシサーバー
- **ローカルデモ**: Node.js（依存パッケージゼロ、標準ライブラリのみ）
- **フロントエンド**: Vanilla JavaScript（フレームワーク不使用）
- **AI**: 商用LLM API（コスト・速度・品質の観点で選定、ベンダー非開示）

### 構成設計の考え方
- APIプロキシ方式で**APIキーをサーバーサイドに完全秘匿**
- フレームワーク不使用で**軽量＋互換性**を担保
- 本番インフラ構成・使用ベンダー・防御パラメータは**セキュリティの観点から非開示**

## AIとの協働プロセス（このプロジェクトの本質）

### 役割分担

| 担当 | 内容 |
|------|------|
| **松島（人間）** | 事業企画・ターゲット定義・価格設計・技術選定の最終判断・顧客折衝・テスト設計 |
| **Claude Code（AI）** | 実装・セキュリティ設計案・リファクタリング・テストケース提案・ドキュメント整形 |

### 具体例：LLM選定の意思決定プロセス

「このチャットボットで使うLLM、何がいい？」という議論で、
以下の判断プロセスを踏みました。

1. **最初にコスト試算** — 主要なLLM API 3つで想定利用量のコスト比較をAIに依頼
2. **応答品質の実測** — 日本語FAQ応答の精度をサンプル比較
3. **結論**: 「低コスト × 爆速応答 × 品質許容」の最適解を判断
4. **運用ポリシー決定**: モデル更新は**無償アップデート範囲内**でのみ実施し、顧客影響を最小化

**この判断はAIが自動で決めたのではなく、事業制約（価格、納品速度、顧客満足度）を加味して人間が決定**しました。

### 具体例：1行で設置 × 商用セキュリティ の両立

「技術者不在の顧客が1行のコードで導入できる」という商品設計上、
セキュリティは**標準装備**として組み込む必要がありました。
AIと対話しながら、以下の観点で多層防御を設計しました。
（具体的な実装パラメータは本番非公開）

| 項目 | 目的 |
|------|------|
| オリジン制限（CORS） | 顧客ドメイン以外からの呼び出しを遮断 |
| レートリミット | 乱用・DoS・コスト攻撃の抑制 |
| 入力長制限 | トークン浪費型攻撃の抑制 |
| プロンプトインジェクション対策 | 役割逸脱・機密情報開示要求の拒否 |
| HTMLタグサニタイズ | XSS等のクライアント側攻撃の遮断 |
| APIキー秘匿 | キー情報をクライアントに渡さない構成 |
| エラー情報隠蔽 | 内部構造の情報漏洩防止 |

AIが提案したコードを、**人間が設計したテストケース**で全パターン検証してから本番投入しました。

## 品質・保守性・リスク管理

一人開発者として「属人化・品質均質性・セキュリティ」に最も向き合っており、
以下の仕組みで担保しています。

### 品質担保
- **AIクロスレビュー**: 実装に使ったAIとは別モデルでコードレビューさせる
  （Claude / ChatGPT / Gemini の複数モデル活用によりバイアスを分散）
- **テスト先行設計**: 機能・セキュリティのテストケースを先に設計し、それを満たす実装を書く
- **実デプロイ前の多段検証**: ローカル → ステージング → 本番の段階的検証

### 保守性担保
- **コードと同時にドキュメント生成**: README・アーキテクチャ決定記録をAIで半自動化
- **ワークフローの標準化**: 受注〜納品をPhase 0〜7で文書化（`docs/workflow.md`参照）
- **プロンプトスキル体系化**: トリガー・入出力・品質基準を定義し、仕組みとして引き継ぎ可能な形に

### リスク管理（自覚している課題含む）
- ✅ **やっていること**
  - 運用ポリシーの文書化（LLMバージョン管理・顧客影響の最小化）
  - 発注前ヒアリングで要件を整理してから実装（手戻り削減）
- ⚠️ **一人開発の限界として自覚しているもの**
  - 人間のシニアエンジニアによる第三者レビューが入らない構造
  - DAST（動的セキュリティテスト）の自動化が未導入
  - 継続的脆弱性監視ツール（Dependabot/Snyk）の本格運用は未整備

**これらは「チーム開発・体系的なレビュー文化のある環境」で補完したい領域として認識しており、業務委託・雇用の場面で最も学びたいポイントです。**

## セットアップ（ローカルデモ）

### 必要環境
- Node.js 18+ （依存パッケージゼロ、標準ライブラリのみ使用）
- LLM API キー（本デモでは `.env.example` を参照）

### 手順

```bash
# 1. クローン
git clone https://github.com/coco888369/asurich-ai-chatbot-demo.git
cd asurich-ai-chatbot-demo

# 2. 環境変数設定
cp .env.example .env
# .env を編集してAPIキーを設定

# 3. 起動
node server.js

# 4. ブラウザでローカルURLを開く
```

## 事業としての設計思想

このプロダクトは技術ありきではなく、
**「中小企業・個人事業主の問い合わせ対応を、手の届く価格で解決する」** という事業課題から出発しました。

- 顧客の作業は「ヒアリングシートに10分回答」のみ
- FAQ作成・プロンプト設計・設置まで当社で完結
- AIが学習するのではなく、**System Prompt にFAQ情報を構造化して記述**（機械学習ではなく仕組みで解決）
- 買い切り型で月額ゼロ → ランニング負担を下げる
- セキュリティは標準装備 → 顧客にリスクを負わせない

この設計思想は[`docs/workflow.md`](./docs/workflow.md) と [`docs/hearing-sheet.md`](./docs/hearing-sheet.md) に実運用フォーマットとして落とし込まれています。
